# Phase 03.5: Pegar Como Humano - Research

**Researched:** 2026-04-27
**Domain:** WhatsApp Web text injection / Lexical contenteditable
**Confidence:** MEDIUM-HIGH

## Summary

WhatsApp Web migrated to Meta's **Lexical** editor (same family powering FB/Messenger/IG). Lexical does NOT trust `KeyboardEvent` for text mutation — it intentionally ignores `keydown/keypress` for content changes and relies almost exclusively on the **`beforeinput`** event (Input Events Level 2) to mutate state. `execCommand('insertText')` still works because internally browsers translate it into a trusted `beforeinput` with `inputType: "insertText"`, which Lexical handles natively. `KeyboardEvent` synthesis (option 1) is the classic trap: text appears via default contenteditable behavior on some paths but Lexical's reconciler reverts it, OR the visual text exists outside the Lexical node tree → pressing Enter sends an empty message. `[VERIFIED: lexical/packages/lexical/src/LexicalEvents.ts]`

**Primary recommendation:** Use `document.execCommand('insertText', false, char)` per character with the editable focused and a valid `Selection` collapsed at the end. Fallback to dispatching a synthetic `beforeinput` + `input` pair with `inputType: 'insertText'` if execCommand becomes a no-op.

## Recommendation

### Method primary (HIGH confidence)
**`document.execCommand('insertText', false, char)`** invoked per-character inside the focused Lexical editable.

Why: Browsers translate `execCommand('insertText')` into a synthetic-but-trusted-internally `beforeinput` event that Lexical's `LexicalEvents.ts` listener handles via its `insertText` command path. Preserves undo stack. Works in 2026 in Chromium despite "deprecated" label — no removal scheduled. `[CITED: developer.mozilla.org/en-US/docs/Web/API/Document/execCommand]`

Critical pre-conditions:
1. Editable must be **focused** (`editable.focus()`)
2. A valid `Selection` must exist (collapsed range at end of last text node)
3. Must be invoked from a **user-gesture-rooted callstack** (Ctrl+Space hotkey handler qualifies; setTimeout-chained callbacks DO NOT — see landmines)

### Fallback (MEDIUM confidence)
If `execCommand` returns `false` or no text appears, dispatch synthetic `InputEvent`:

```js
editable.dispatchEvent(new InputEvent('beforeinput', {
  inputType: 'insertText', data: char, bubbles: true, cancelable: true
}));
editable.dispatchEvent(new InputEvent('input', {
  inputType: 'insertText', data: char, bubbles: true
}));
```

Caveat: `event.isTrusted=false`. Lexical's listener does NOT gate on `isTrusted` (verified in source) so this works for Lexical specifically — but WA may add an isTrusted check at the React layer. Test before shipping.

### Methods to AVOID
- **`KeyboardEvent` synthesis** (keydown/keypress/keyup): Lexical ignores keypress for content; only uses keydown for shortcuts/navigation. Will NOT mutate the editor state. `[VERIFIED: LexicalEvents.ts]`
- **Direct Lexical internals access** (`__lexicalEditor` on DOM node): Brittle, version-coupled, unminified-symbol-dependent. WA ships minified — symbol names change every release.
- **`textContent =` / `innerText =`**: React/Lexical reconciler reverts on next tick; Enter sends empty.

## Pseudo-code (planner-ready)

```js
async function humanType(text) {
  const editable = document.querySelector('div[contenteditable="true"][role="textbox"]');
  if (!editable) throw new Error('compose not found');
  editable.focus();
  // Place caret at end
  const range = document.createRange();
  range.selectNodeContents(editable);
  range.collapse(false);
  const sel = window.getSelection();
  sel.removeAllRanges(); sel.addRange(range);

  for (const ch of text) {
    const ok = document.execCommand('insertText', false, ch);
    if (!ok) {
      // fallback: dispatch beforeinput
      editable.dispatchEvent(new InputEvent('beforeinput',
        {inputType:'insertText', data:ch, bubbles:true, cancelable:true}));
    }
    await sleep(50 + Math.random()*100);
    if (/[.,;!?]/.test(ch)) await sleep(150 + Math.random()*200);
  }
}
```

## Landmines (HIGH confidence)

1. **The "ghost text" bug**: Text visible in DOM but Lexical state empty → Enter sends empty / nothing. Detect by reading `editable.textContent` AFTER injection AND attempting send; if message bubble doesn't appear in chat list within 2s → injection failed Lexical state.
2. **User-gesture loss**: Chaining `setTimeout(..., 50)` between chars breaks the user-activation chain. `execCommand` may silently no-op. Mitigation: the *first* char of the burst MUST run synchronously inside the hotkey handler. Subsequent chars work because Lexical doesn't require gesture for `insertText` once focus is established — but verify in target Chrome version.
3. **Selection collapse**: If you rebuild Selection between chars, you can land in the wrong node. Don't touch Selection after the initial setup; let `insertText` advance the caret.
4. **Composition events**: If user has IME active, synthetic insertText interferes. Detect with `editable.matches(':composing')` (or track `compositionstart/end`) and abort.
5. **Multi-line `\n`**: WA Web treats Enter as send. To insert newline use Shift+Enter equivalent: `execCommand('insertLineBreak')` or `insertText('\n')` with caveats — test.
6. **Send button vs Enter**: After typing, prefer **clicking the send button** (`[data-testid="send"]` or `[aria-label*="Send"]`) rather than dispatching Enter — more reliable, sidesteps key-event-trust issues for the send action.

## Anti-bot / isTrusted

No public evidence WA Web checks `event.isTrusted` on input events as of late 2025 (existing extensions like WA Web Plus, quick-reply, Whatsappium use synthetic events without obfuscation). `whatsapp-web.js` uses Puppeteer (which produces trusted events) so it's not a comparison. **LOW confidence** that this stays true — Meta could add detection. Mitigation: use `execCommand` as primary since it produces internally-trusted events.

## How to test (5 min)

1. Open `web.whatsapp.com`, open any chat, focus compose, open DevTools console.
2. Paste the `humanType` function, then run `humanType('hello world')`.
3. **Pass criteria**: characters appear one by one AND clicking the native send button (or Enter) actually delivers the message (it appears as a chat bubble with "✓✓").
4. **Fail signature**: text appears but Enter does nothing / message bubble missing → Lexical state not updated → fall back to InputEvent dispatch path.
5. Sanity check on state: `document.querySelector('[contenteditable=true]').textContent` should equal what you typed.
